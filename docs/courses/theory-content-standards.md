# Theory content standards

Короткий стандарт для теории Uchicode, задач, подсказок, частых ошибок и guide-страниц.

## Область

- Работать только с открытыми разделами, если задача явно не просит новую теорию.
- Не дописывать полностью закрытые разделы и разделы `needs-theory` во время точечной полировки.
- Сохранять текущий уровень курса: база C++ остаётся базой, ООП-разделы фокусируются на текущей ООП-теме.

## Структура раздела

Хороший теоретический раздел должен дать ученику:

- что означает понятие;
- зачем оно нужно в текущей теме;
- один минимальный компилируемый пример;
- разбор примера;
- одну-две частые ошибки;
- направление для связанной практики.

Заголовки не обязаны совпадать во всех файлах, но эти смыслы должны быть закрыты.

Content-файл раздела задаёт только учебные данные и markdown-секции. Layout для header, common mistakes, self-check, previous/next navigation и “Задачи после темы” задаёт общий renderer страницы.

## Примеры кода

- Держать примеры короткими и сфокусированными на одной идее.
- Предпочитать компилируемый код, если блок явно не помечен как ошибочный пример.
- Не показывать undefined behavior и небезопасные приёмы, если урок не разбирает именно эту ошибку.
- Использовать понятные имена переменных, полей, методов и классов.
- Не добавлять в базовые разделы templates, move semantics, smart pointers, exceptions или внутренности STL, если текущая тема этого не требует.
- В новых примерах предпочитать явный `std::`. Оставлять `using namespace std;` только там, где окружающий урок сознательно использует это для начальной читаемости.

## Частые ошибки

- Добавлять ошибки, в которые ученик реально может попасть в текущей теме.
- Объяснять причину и исправление, а не только симптом.
- Не раздувать каждый урок длинным общим списком ошибок.
- Держать task-specific ошибки внутри задачи, а повторяющиеся советы по обучению — в guide или документации.
- В theory content использовать обычный раздел `## Частые ошибки`; визуальную структуру с eyebrow `Разбор ошибок`, title `Частые ошибки`, карточками и `CodeBlock` задаёт общий renderer.
- Не создавать локальный формат ошибок внутри отдельной темы.

## Самопроверка

- Использовать раздел `## Мини-проверка` или `## Самопроверка` со списком вопросов.
- Не размечать self-check вручную: общий renderer превращает список в интерактивную проверку с eyebrow `Самопроверка` и title `Мини-проверка`.
- Вопросы должны быть короткими и проверять текущую тему, а не будущие разделы.

## Связь теории и задачи

- Теория должна готовить к доступным связанным задачам.
- Если задача требует понятие, которого нет в разделе, добавить короткое объяснение в ближайшую релевантную тему.
- Если теория объясняет то, что пока не нужно, сократить это или перенести в будущий раздел.
- Не дублировать список practice-файлов вручную в конце теории, если CoursePage уже показывает системный блок “Задачи после темы”.
- Практические файлы и стартовый код показываются на странице конкретной задачи, а не отдельным ручным списком в теории.
- Если у темы нет связанных задач, CoursePage показывает короткий системный empty state; не добавлять вместо него локальные placeholder-списки.

## Источники

- Использовать cppreference для проверки синтаксиса и поведения языка.
- Использовать C++ Core Guidelines для проверки современных практик и поддерживаемости.
- Использовать качественные tutorial-материалы только как ориентир по подаче.
- Использовать форумы и Stack Overflow только для поиска типичных ошибок новичков.
- Не копировать длинные фрагменты внешних текстов.
- Не выдавать непроверенные утверждения о C++ за факты.
