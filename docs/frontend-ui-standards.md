# Frontend UI standards

Короткий стандарт для учебных страниц Uchicode. Он нужен, чтобы задачи, теория и справочные страницы выглядели как один продукт.

## Типографика

- Font stack: системный sans-serif без CDN и внешних загрузок.
- Body: базовый размер `1rem`, `line-height` около `1.6`.
- Учебный текст: ширина строки около `68-76ch`, без плотных абзацев.
- `h1`: главный заголовок страницы, заметный, но без hero-масштаба внутри рабочих страниц. Для рабочих страниц держать примерно `1.5-1.875rem`.
- `h2`: заголовок смыслового блока. В panel он не должен конкурировать с page `h1`.
- `h3`: подзаголовок внутри блока, без случайной жирности.
- Metadata/caption: компактный второстепенный текст, не использовать как основной контент.
- Inline code: только для синтаксиса, имён файлов, команд и коротких C++ фрагментов.
- Пользователь не должен видеть обратные кавычки. Строки с `` `code` `` нужно рендерить через inline-code.
- Inline-code остаётся частью предложения: `display: inline`, без block/grid-обёртки вокруг самого текста.
- Учебные страницы используют спокойную типографику: body text и descriptions не bold, panel heading умеренный, line-height читаемый.
- Для новых UI-зон использовать спокойные веса: `500/600/650`; `700` только для сильного display-акцента.
- `font-weight: 800/900` не использовать в обычных страницах, карточках, meta, descriptions, badges и landing-блоках.

Запрещено:

- случайные размеры шрифта и случайная жирность;
- uppercase для обычного текста;
- декоративные символы без функции;
- длинные пояснения там, где достаточно короткой инструкции.
- общий план решения, скопированный в каждую задачу.

## Units

- Использовать `rem` для масштабируемых UI-размеров: текст, отступы, radius, размеры компонентов, grid `minmax` и breakpoints.
- Оставлять `px` для `1px` borders, hairline dividers, shadows, outlines и намеренных pixel corrections.
- Не делать слепой `px` -> `rem`; спорные значения оставлять как есть.
- Новые стили должны использовать существующие переменные и tokens проекта.

## Компоненты

- `BackLink` — единая ссылка назад. Не оставлять голый текст вроде “К задачам”.
- `MetaRow` / `MetaItem` — курс, сложность, файлы, статус и другие параметры страницы.
- `StatusBadge` — только для состояния, не для сложности.
- `.count-badge` — единый спокойный счётчик для количества задач, уроков, найденных ошибок и пунктов.
- `EmptyState` — контекстное пустое состояние на ширину списка или контентной области: на desktop текст слева, действие справа; на mobile одна колонка, действие под текстом.
- `TaskActionBar` — основное действие на странице задачи.
- `CollapsibleSection` — единый раскрывающийся блок.
- CTA-ссылки и кнопки должны идти через shared action components: `Button`, `LinkButton`, `ActionButton`.
- Meta badges должны быть компактными и стабильными: одинаковая высота, нормальный line-height, label второстепенный, value без лишней жирности.
- Empty states должны давать короткое объяснение и один основной следующий шаг, без случайного центрирования кнопки.
- Labels для сравнений вроде “Ошибка / Правильно” должны быть compact badges, а не крупные заголовки над кодом.

## Inputs

- Все search/input/select должны иметь одну систему: radius около `0.625rem`, спокойный border, `surface-2` background, видимый focus ring.
- Placeholder не должен выглядеть как disabled-текст.
- Search-блоки не делать резкими прямоугольниками; они должны совпадать с остальными формами.
- Для login/register/profile можно менять размер, но border, radius и focus logic должны оставаться той же системы.
- У `select` нельзя менять shorthand `background` на hover/focus, если стрелка нарисована через `background-image`. Менять только `background-color`, border и focus ring.
- Dropdown/select для фильтров должны использовать общий компонент или общий стиль: `cursor: pointer`, стабильный hover/focus/selected state, список в стиле продукта, без системного Windows-select.
- Native `select` допустим только там, где раскрытый список визуально не выбивается из интерфейса.

## Auth pages

- Login/register используют общий выразительный auth layout: форма основная, companion panel второстепенный.
- Companion panel должен быть полезным и заполненным, а не пустой карточкой или голым списком.
- Login/register используют shared form/button styles, без отдельной системы CTA.
- Labels всегда видимы; placeholder не заменяет label.
- Auth CTA должен быть понятным, контрастным и стабильным на hover/focus/disabled.
- Helper panel второстепенный: короткий список, без marketing-copy и тяжёлых заголовков.
- Autocomplete: login `username` / `current-password`, register `username` / `new-password`.
- Theme checks для auth выполнять risk-based: формы, inputs, CTA, helper panel и topbar auth links.

## Buttons

- Hover не меняет размер, padding, позицию или контраст текста.
- Не использовать `translateY` на hover: карточка или кнопка не должна “уезжать” от курсора.
- Иконки, стрелки и chevron внутри button/select/filter не меняют позицию на hover.
- Chevron вращается только по состоянию open/closed, а не при обычном наведении.
- Focus ring должен быть видимым и аккуратным: `outline` или `box-shadow`, без двойной случайной обводки.
- Disabled выглядит недоступным, но текст остаётся читаемым.
- Active state может быть спокойным затемнением, без сдвига layout.

## Home page

- HomePage для `/` — публичная главная без рабочего app-shell: без sidebar, search, topbar и AI assistant.
- Главная объясняет первый маршрут новичка: первый шаг, первая тема, простой пример, частая ошибка, первая задача.
- Главный visual — карта маршрута, а не code editor, dashboard или сетка одинаковых карточек.
- Hero `h1` держать примерно до `3rem` на desktop; section title — примерно `1.4-1.75rem`.
- На mobile не растягивать первый экран искусственно: пользователь должен быстро увидеть следующий полезный блок.
- CTA не должны повторяться внизу страницы, если они уже есть в hero и в секции выбора маршрута.
- Каждый блок показывает конкретное действие пользователя: открыть курс, понять формат урока, выбрать маршрут или перейти к задачам.
- Тексты короткие и без абстрактных обещаний: не использовать рекламные слоганы, жаргон и claims без функции.
- Анимация на главной допустима только если она помогает понять маршрут; `prefers-reduced-motion` обязателен.
- Кликабельные карточки на главной должны быть понятны до hover: видимое action label, стрелка или button-like footer.
- Некликабельные info-карточки должны быть спокойнее: без ссылочного hover, без `cursor: pointer`, без action footer.
- Карта маршрута не должна быть сеткой одинаковых карточек: использовать линию пути, короткие станции и отдельную active-card.

## Карточки и секции

- Карточка должна иметь одну главную цель: открыть курс, тему или задачу.
- Статус в карточке всегда в одном предсказуемом месте.
- Не смешивать статус, сложность и количество файлов в одну россыпь бейджей.
- Большие секции без практической пользы удалять или сокращать.
- Между последней карточкой и следующим divider/section должен быть явный вертикальный интервал.
- Divider не должен слипаться с карточкой, списком, код-блоком или следующим заголовком.
- Empty/loading/error states должны прямо говорить, что произошло и что сделать дальше.

## Task cards

- Status badge в карточке задачи закрепляется в отдельной правой колонке и не сжимается из-за заголовка.
- Title занимает только доступное место слева; длинные названия переносятся или clamp-ятся без выталкивания статуса.
- Карточка задачи имеет стабильные зоны: header, meta, tags, action.
- Compact meta в карточке задачи не пишет сырой текст `Label: value`. Для сложности и файлов использовать короткие value-only chips или явную label/value-разметку.
- Meta, tags и action не должны менять вертикальную структуру карточки от количества тегов или длины title.
- Action button находится в одном визуальном месте и не меняет размер на hover/focus.
- Hover/focus карточки, badge, tags и action не меняют padding, высоту, ширину и не двигают соседние элементы.
- Tags переносятся внутри своей зоны и не выталкивают action-зону.

## Meta и счётчики

- В compact chip не использовать формат `Сложность: базовая` или `Файлы: 1 файл`, если label и value визуально слиплись в одну строку.
- Если label нужен для понимания, делать definition-style: label второстепенный, value основной, без случайной жирности и uppercase.
- Count badge использовать для одинакового смысла: найдено всего, количество задач/уроков в секции, найденные ошибки, пункты в collapsible.
- Section header с количеством строится как `h2` плюс `.count-badge`, а не как обычный серый текст.
- Count badge должен быть заметным, но спокойным: без ярких статусов, без изменения размера на hover/focus.
- Для маленьких зон вроде collapsible header использовать компактный вариант `.count-badge--compact`.

## Collapsible

- Header всегда `button`.
- `aria-expanded` обязателен.
- Focus state должен быть видимым.
- Chevron — CSS/иконка, `aria-hidden`, без сырого символа `▶` как текста.
- Счётчик вроде “3 пункта” использовать через `.count-badge`, только если он помогает оценить объём.

## Step cards и списки

- Step-card использует стабильную сетку: фиксированная колонка номера и отдельная колонка текста.
- Номер шага выровнен одинаково во всех карточках и не “плавает” рядом с контентом.
- Заголовок и описание в step-card занимают фиксированную верхнюю зону на desktop, чтобы списки действий не начинались на случайной высоте.
- Внутренний список действий не превращать в набор декоративных бейджей. Если это обычные действия, использовать спокойные маркеры.
- Guide step-card не должен быть тяжёлой пустой карточкой: короткий header, конкретный список действий, без декоративных панелей без функции.
- Блок “Если застрял” должен быть частью той же системы: компактная секция, конкретные проблемы и действия, без отдельного визуального жанра.
- Нумерованные списки нужны только там, где порядок реально важен.
- На mobile номер, заголовок и пункты должны оставаться в одной читаемой сетке без горизонтального скролла.

## Task header и meta row

- Header задачи — компактный цельный блок: `BackLink`, номер/раздел, `h1`, `MetaRow`, затем `TaskActionBar`.
- `h1` на рабочей странице не должен быть hero-размера. Если заголовок спорит с контентом, уменьшить его до спокойного page-title.
- `BackLink` не должен менять padding, ширину или высоту при hover/focus.
- `MetaRow` не выводится как серая строка plain text и не растягивается в крупные карточки. Использовать компактные `MetaItem`: label второстепенный, value основной.
- Статус внутри `MetaRow` показывать через `StatusBadge`, но рядом должен быть текст статуса, а не только цвет.
- Labels в meta row не делать uppercase по умолчанию.
- На mobile meta row переносится чипами без горизонтального скролла.
- TaskDetailsPage держит структуру: header, meta, progress/action, условие, ожидаемый результат, файлы/code.

## Status/action без скачков

- `TaskActionBar` должен иметь стабильную структуру во всех состояниях: in progress, solved, loading, error.
- `TaskActionBar` не должен выглядеть как большой баннер; высота и padding минимальны для действия и статуса.
- Не показывать и не скрывать целые строки так, чтобы нижние секции прыгали. Для сообщения резервировать минимальную высоту.
- Кнопка действия не должна менять ширину layout-контейнера при смене текста.
- Success/error message размещать в одной зоне, с `aria-live`, без вывода traceback или сырых ошибок.
- Перед merge вручную проверить: нажать “Отметить решённой” и “Вернуть в работу”; контент ниже не должен дёргаться.

## Страница задачи

TaskDetailsPage строится как учебный flow: задача -> файл/code -> помощь -> теория. Основной контент важнее helper-блоков; helper sections в collapsed-состоянии должны быть компактными и не перетягивать внимание. Новые задачи добавляют только данные в `tasks.ts`; page layout не дублируется в тексте задачи.

Не повторять общий методический текст в каждой задаче. Если нужен общий порядок решения, давать компактную ссылку на методику. Блок теории использовать как "Тема для повторения" с действием "Открыть раздел"; не использовать двоеточия в button labels. Panel text не должен быть bold по умолчанию.

Повторяемый helper text должен быть компактным: не дублировать длинные пути и инструкции рядом с уже показанным файлом или code block.

На странице задачи должно быть видно:

- куда вернуться;
- название и раздел;
- курс, сложность, файлы и статус в `MetaRow`;
- основное действие в `TaskActionBar`;
- условие;
- файлы и каркас;
- подсказки, частые ошибки и самопроверка в одинаковых collapsible-блоках.

Общий план решения не дублировать в каждой задаче. Если план типовой, оставить компактную ссылку на “Как учиться”. Уникальные шаги задачи можно показывать отдельно.

Inline-code внутри списков и чеклистов не должен распадаться по буквам. Короткие фрагменты вроде `push_back`, `size()` и `empty()` держать цельными; длинные пути переносить отдельно через path/meta-стиль без горизонтального скролла.

File/code panel не должен растягиваться в большую пустую область. Если стартового кода нет, показывать компактное пустое состояние; practice path держать рядом с meta файла и переносить без поломки layout.

Theory pages Базы C++ и ООП C++ используют одну структуру: page header, theory content, previous/next navigation, системный блок “Задачи после темы”. Content-файлы дают данные и markdown-секции; layout живёт в общем renderer. Списки practice-файлов не дублировать вручную в теории, если задачи уже выводит общий блок.

Common mistakes на theory pages выводятся только через единый pattern: eyebrow “Разбор ошибок”, title “Частые ошибки”, компактные карточки “Ошибка N”, единый `CodeBlock` для примеров с кодом.

Self-check на theory pages выводится через общий интерактивный компонент: eyebrow “Самопроверка”, title “Мини-проверка”. Заголовки `Мини-проверка` и `Самопроверка` в content должны давать этот UI, без ручных локальных списков.

Блок “Задачи после темы” показывает реальные task cards, если задачи связаны с разделом. Если задач нет, используется compact empty state с коротким текстом, без placeholder-карточек и ручных practice-списков.

## Как учиться

Страница “Как учиться” хранит общую методику:

- как читать условие;
- как начать решение;
- как собрать минимальный рабочий вариант;
- как проверять результат;
- что делать при ошибке компиляции;
- как пользоваться подсказками;
- когда отмечать задачу готовой;
- что делать, если застрял.

Текст должен быть коротким и практичным. Без мотивационной воды.

Проверка текста:

- каждая фраза отвечает на вопрос “что сделать?”;
- нет общих банальностей вроде “старайся лучше”;
- нет повторов из конкретных задач;
- общие инструкции живут здесь, а не в каждой задаче.

## Чеклист перед merge frontend

- `npm run typecheck`
- `npm run lint`
- `npm run build`
- нет `/api/api` в `site/src` и `site/dist`
- нет сырых backticks в пользовательском UI: backticks в source допустимы только как markdown/input для renderer или как TypeScript-синтаксис
- нет `site/dist`, env-файлов и временных файлов в git
- статусы не объясняются только цветом
- keyboard focus виден
- mobile layout без горизонтального скролла
- theme checks выполнять risk-based: проверять только затронутые компоненты и состояния в существующих темах
